V2 data spec: fleet, global scale, returns & FMPs

Extends the v1 spec (built, benchmarked, conclusions standing) with the four review points: factor returns and factor-mimicking portfolios in the daily load; a global model size class; a ten-model vendor fleet with custom variants; serial query chains. Every dimension below is a number a practitioner can eyeball — lines like this are the explicit validation asks, consolidated in §6. All v1 mechanics carry over unchanged (deterministic seeds, AR(1) persistence, sparse loading storage, per-model conventions, PSD covariances).

Global factor split — is 16 style / 64 industry right for the 81 non-country/currency factors of AXWW4-MH? And is this region/vendor mix representative?

A variant is a new model_id generated from a base ± a factor delta, fully materialized: shared factors byte-identical to the base by seed construction, custom styles fresh AR(1) series. Nonzero loadings per asset-day: globals ~20 (16 styles dense, 1–2 industry, 1 country, 1 currency, market), regionals ~15–16. Multi-listed securities carry exactly one country and one currency loading. OK to ignore multi-currency lines / DR-vs-ordinary linkage for a storage benchmark? Variant cadence — appears ~quarterly, lives for years?

3 · Storage strategies — the table-per-model question, measured

The review's operational criterion: adding a model must never become a project. Three physical strategies, one normalized source, all benchmarked — plus an operational drill (add a model, add a variant) scored on wall-time, steps, and bytes rewritten.

strategyadd a model costsobjects at 12 modelsexpected weakness
Aper-model wide tables (v1)config + transform run (DDL generated)24+ tablesobject count compounds
Bone generic-slot table + alias map + generated viewsinsert map rows + create views1 table + viewsgovernance; wide footers
Cnormalized longnothing1 tablepivot cost (measured: 5–48×)
 
wide_generic(
    model_id, cob_date, asset_id, version_id, specific_risk,
    F001 DOUBLE, F002 DOUBLE, ... F260 DOUBLE        -- slot pool ≥ max model + headroom
)
factor_slot_map(model_id, slot_id, factor_id, factor_seq,
                valid_from, valid_to)                 -- temporal; slots append-only, never reused

CREATE VIEW wide_cs_barra_use4_l AS                   -- generated; the ONLY query surface
  SELECT cob_date, asset_id, F001 AS "BETA", F002 AS "MOMENTUM", ..., specific_risk
  FROM wide_generic WHERE model_id = 'BARRA_USE4_L';

What makes B safe: humans never query Fnnn columns — the generated views restore loud failure (missing named column) on wrong-model access, instead of silently returning another factor's numbers. Partitioned by model_id, per-model scan behavior should match A; unused slots are NULL (near-free in Parquet) but widen footers — one of the things measured. A float-only slot pool suffices for this payload.

6 · Validation questions, consolidated

  1. Global factor split: 16 style / 64 industry / 95 country / 72 currency / 1 market for the AXWW4-MH proxy?
  2. Coverage 58k / estu 13k global; regional universes per the §1 table?
  3. FMPs: estimation-universe-only, dense weights, styles + market only?
