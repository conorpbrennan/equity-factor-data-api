The headline: materialize for the query, keep one source of truth

The single most consequential measurement in the exercise.

Factor data is written once a day and read all day, in two shapes: cross-sections (one date, many assets — risk runs, screens) and time series (few assets or factors, many dates — history, drill-downs). Storing the data normalized (long-form) is right for correctness, restatements, and lineage — but querying it directly means re-pivoting on every read. The scaffold keeps the normalized store as the source of truth and materializes two query-shaped wide layouts from it: date-major for cross-sections, asset-major for time series. Measured on the largest model (248 factors, 58,000 assets, 20 years — 302M rows per layout):

Warm p50, full 12-query suite in the repo; the materialized layouts win every one of the twelve, cold and warm. Point-in-time needs no third copy: the wide layouts hold originally-published values and an overlay join pulls restated rows from the normalized store — on a date with 1.2M republished values the overlay answer is identical to the from-scratch answer, in 0.3s versus 27s.

Adding a model must never become a project.

The wide layouts use one generic-column physical schema (F001…F260) with a mapping table and generated per-model views, rather than one table per model. Measured at full scale this costs nothing — it matches per-model tables on every query — and adding a model or a custom variant becomes mapping rows plus a view, not new tables and data movement. Chris flagged model customization as a strong desire that doesn't exist today; this design makes a variant cheap by construction.

The dataset: sized from Chris's numbers, larger than today's estate

The store was built to the dimensions Chris provided from SFM practice, then extended to stress the design: 12 models — ten vendor models across US/EU/JP/EM in both Barra-style and Axioma-style conventions, including two global models (the AXWW4-MH proxy: 248 factors, 58k coverage / 13k estimation universe), plus two custom variants of the kind SFM wants but doesn't run yet. Twenty years of daily history with factor returns, factor-mimicking portfolios, restatements, and a same-day estimate stream — about 680 GB. Chris's reference point is 5–10 vendor models in practice, so this deliberately exercises more models, more history, and bigger universes than SFM carries today. Generation is deterministic: built twice from the same config — laptop and AWS — the files came out byte-identical, so any finding can be reproduced from a one-page config.

On technology: formats are a commitment, engines and platforms are not.

Parquet is the storage format — open, columnar, the de-facto industry standard, readable from every stack, and a format I've run in production for years. That choice should be uncontroversial. DuckDB appears here only as an open-source query engine used as a measurement instrument — it made the benchmarks cheap to run locally and in-region. Nothing in this work proposes it (or any engine) as SFM's stack; the layout findings and timings above are properties of the data organisation and transfer to whatever engine the data teams choose. The one catalog effect measured — query planning that never touches file metadata — is equally available from any open cataloged table format (Iceberg-class or similar).

The sizes put the platform question in proportion: the full daily vendor load across all twelve models is roughly 130 MB, the twenty-year store is 680 GB in total, and a user's interactive working set is megabytes. At these volumes query performance is decided by data organisation — it is already saturated on commodity hardware, as the table above shows — so a platform choice can be weighed on the things platforms genuinely differentiate on: governance, entitlements, cataloging, operations, and sharing. Any platform that reads open formats can adopt these layouts unchanged, which makes the findings here an input to that evaluation rather than a competitor to it.

Architecture

Two layers with two contracts: a lenient facade humans actually use, and a strict core library that performs no I/O — every input is passed in, validated, exact. The facade is the only component that loads: covered requests are answered from its warmed working set locally, and a miss sends SQL to the on-demand in-region engine, which scans the Parquet next to the data and shuts down when the queries stop. With no engine configured the facade scans S3 directly — slower, same results. Loading, unit conversion and id resolution therefore live in one place, and the core stays pure enough for any system to build on.

Design principle: the core computes; it never loads.

All data enters the core library by being passed in — validated, exact, canonical. It performs no I/O of its own. That buys three things: the core is deterministic and trivially testable (no store, no network, no clock in the way); there is exactly one load path in the system, so unit conversion, identifier resolution, and any future entitlement or audit control happen in one place rather than wherever a query happened to run; and the core is portable — it computes the same answer whether the data arrived from S3, a warmed local set, or a future platform's tables. The strict/lenient split only works because of this: leniency lives where the loading is, purity lives where the maths is.

Where compute belongs: a requirement, measured

Bulk work near the data, once; interactive work on the user's machine, all day.

One usage requirement fell out of measurement rather than preference. The store lives in AWS Ireland by design: the client-to-store distance from a New York laptop mirrors what a London user would experience against a US-region store. Most of SFM is US-based and would sit in-region with the data, so the cross-Atlantic numbers here are a conservative upper bound — the worst seat in the house, not the typical one. Scanning the store across that distance ran a twelve-question scripted session in 1,099 s; routing the same scans to a small on-demand node in-region halved it to 546 s, and a repeated scan dropped a further ~10× (2.6 s → 0.26 s) served from the engine's own in-memory cache — server-side, shared by all users while the node is up. So scans belong next to the data. But the larger effect runs the other way: a scheduled morning job — run centrally per desk in a deployed setup; a cron entry on the user's machine in the scaffold — warms the day's working set from the position list, once, and persists it. Every covered question after that is answered on the user's machine, in-process, in milliseconds — whole sessions that never contact the store at all. Risk users work in notebooks with local DataFrames; an access layer that pushes their interactive loop to remote compute fights how they actually work. The design conclusion: push the heavy scans to the data once a day, then keep the day local.

What runs on top

The facade is far more than the loader. Everything a user needs and a strict core must refuse lives in this one layer, so it is defined once, identically for every consumer:

  • leniency at the boundary — dates as strings, 'latest', or proper types; assets as internal ids or any vendor scheme, resolved through the dated cross-reference; unknown ids fail loudly rather than silently dropping from results
  • one set of canonical units — Barra and Axioma publish returns and risk in different raw conventions; the facade converts exactly once, on the way out, so no two consumers can ever hold different numbers for the same quantity
  • notebook-shaped output — wide frames (one column per factor), pandas by default per the installed base, polars as an opt-in; conversion happens at the return boundary only
  • discoverability — list_models(), describe(), styles vs industry factors, date ranges, raw vendor conventions: orientation without tribal knowledge
  • working-set lifecycle — warm from a position list, persist, cold-start offline, 1-day TTL, explicit clear() on a restatement: cache behaviour is policy in one place, not folklore per notebook

Analytics, stateless over the core:

  • portfolio exposures and active-vs-benchmark tilts
  • PnL attribution, with flash PnL the same decomposition on a same-day estimate stream (FMP weights × asset returns) — the flash-to-official revision is only the return stream swapping underneath
  • exposure-change drill-down: which factor moved, then which assets drove it
  • a batch runner that computes and persists all of this per portfolio, cron-ready

Conventions as code:

  • column names as shared constants; legacy naming sanitized at the boundary
  • unit conventions as executable conversions — Barra and Axioma publish in different raw units; the facade returns one canonical set
  • identifier schemes as string constants over a dated cross-reference — vendor ids in, internal ids inside, never chained vendor-to-vendor
  • a static model inventory so model ids are imported, never re-typed

Plain-English access — and a usability test of the API

A one-page skill file teaches an AI assistant the two packages; the assistant writes and runs real code.

The repo carries a Claude skill: a single page of instructions describing the facade and analytics APIs, their conventions, and a handful of worked recipes. With it, a session answers questions like "I hold 10mm of asset 1, 20mm of asset 2, short 5mm of asset 3 — what are my factor exposures today?" or "my momentum exposure moved — where is it coming from?" by writing and executing real snippets against the same API every user has, and answering from the resulting frames. Nothing is recalled or approximated: every number in an answer was computed live, and the snippet that produced it is in the transcript, so any answer is verifiable.

A twelve-prompt scripted session — discovery, data pulls, portfolio analytics, flash-vs-official PnL, the exposure-change drill-down, the cache workflow, and deliberate error cases — ran unattended end-to-end against the full AWS store; the transcript with per-prompt timings ships in the repo, alongside its micro-store twin. The error cases matter most: handed a string date, the session relayed the strict core's exact refusal; asked for an estimate stream on a store that carries none, it reported the precise error rather than quietly substituting official numbers — the fail-loud design surviving contact with a language model.

The deeper value is what it proves about the API itself: if a model can drive the whole layer correctly from one page of instructions and the READMEs, a new joiner can too. The skill doubles as a continuous usability test — every place the assistant stumbles is a place a human would have.

The method is the generic part

Chris's steer: the solution should be generic, not factor-model-specific. Agreed — what generalizes is the method.

Nothing above depends on factors. The sequence is: enumerate the query patterns first; keep one normalized, restatement-carrying source of truth; materialize a layout per dominant query shape; make the physical schema generic — slots plus a mapping table plus generated views — so extending it (a new vendor, a new model variant, a new field) is rows, not DDL or data movement; and put a catalog over the files so query planning is a metadata lookup, not a scan. Factor models made a good demonstration because their shapes are unforgiving — a 250-column cross-section one moment, a twenty-year single-asset path the next — but positions, prices, and sensitivities data share the same read-mostly, shape-polarized character. The slot model is specifically how a schema absorbs the next addition without a migration, which is the property that keeps a generic design generic.

Decided with Chris, and what's open

Settled (implemented):

  • user layer speaks pandas by default — the installed notebook base is pandas-native; polars stays a per-facade opt-in
  • sec_id_type is a plain string with shared constants — odd schemes need no code change
  • cache invalidation: 1-day TTL on persisted working sets (morning re-warm cadence) plus clear() for known restatements
  • model discoverability: static inventory now; target state is the inventory living in the curated store itself
  • conventions governance: firm-wide conventions curated with other teams and ratified by Alex; model-specific ones handled separately

Open, deliberately:

  • canonical units at the core layer vs the facade (Chris leans core; I'm inclined to agree — it's a load-bearing refactor, sketch first)
  • core operating on native vendor security ids (today's FMA practice) vs internal integer ids (as built)
  • ModelFacade ownership — QDS-adjacent or Risk-owned; my lean is Risk, given it is almost entirely a usability layer
  • trading-day count as a per-model property (252 vs 260/261 for some global models)
  • FMP time-series queries want the same dual materialization as loadings, if that access pattern is real
