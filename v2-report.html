The v2 factor store: twelve models, three schemas, two machines, one dataset

Everything in this report rests on synthetic data produced by a purpose-built generator (python_src/genv2). No vendor data is involved: each model in the fleet is a config entry — factor taxonomy, universe regions, coverage rates, vendor conventions, dynamics — and the generator turns that config into 20 years of daily loadings, covariances, specific risk, factor returns, FMP weights, and restatements. Values are engineered to be realistic where realism affects storage behavior: loadings follow AR(1) with cross-sectional standardization (compression matches persistent real data, not incompressible noise), industry/country/currency memberships are near-static one-hots, covariances are PSD by construction, and every number is rounded to vendor-file precision. Generation is deterministic — counter-based seeds keyed by (seed, model, component, date) — so the same config yields byte-identical Parquet on any machine, which §4 exploits as a correctness proof. Adding a model to the fleet is adding a config entry; nothing downstream is hand-built.

V2 scales this to practitioner-reviewed dimensions: a ten-vendor-model fleet plus two custom variants, global models at AXWW4 size, and the full daily data load. The tier was built twice from the same seeds — on this dev box and on EC2 writing directly to S3 — then verified equivalent, and the three candidate physical schemas were benchmarked against each other at global scale, locally and across the Atlantic.

Neither machine was the bottleneck — memory budgets were. The same model's asset-major bucket pass took 9,488 s in a spilling 12 GB lane and 1,269 s in a 45 GB no-spill lane on the same box — 7.5× from memory calibration alone. That result is now codified: lane plans are sized so no lane ever spills (genv2 plan), spill directories are per-process, S3 writes are retried ×8, and destination prefixes are force-emptied before APPEND passes. Every one of those rules was paid for by an incident during these builds.

4Equivalence: one dataset, proven twice

Byte-identity across architectures also transfers all quality properties measured on one store to the other — compression behavior, statistics, layouts. Either copy is ground truth for the other.

5Query performance: three schemas × two locations

Model under test: AX_WW4_MH — 248 factors × 58k assets, 302M wide rows. Four arms: the three physical schemas plus D — DuckLake, a catalog layered over strategy B's generic tables (not a fourth schema: the same bytes, planned from catalog metadata instead of Parquet footers). Ink: the dev box reading its local copy (SATA). Orange: the same box querying the S3 store across the Atlantic (~75 ms RTT). Cold = fresh process; warm = repeats in-process. p50 seconds.

Reading cold vs warm — what is cached, and what never is

Results are never cached: every warm repeat re-executes the full query — scan, decompress, filter, join, sort, Arrow, Polars. What warm reuses is bytes: locally the OS page cache; remotely DuckDB's in-process range cache, so warm remote repeats move no data across the Atlantic. One S3 cost survives warm for the raw-Parquet arms: file discovery (glob/LIST) re-executes per query — that API traffic is the ~0.5–1.4 s floor under A/B's remote warms, and its absence is why D (catalog-planned, zero S3 calls at plan time) runs warm-from-New-York at local-disk speed.

Cold is the uncached truth: a fresh process each time — page cache evicted locally, the range cache dead by construction remotely (it is per-process; no disk cache is used in these grids). Read warm as "a session re-querying data it already touched", cold as "first touch". One-shot batch workloads live in the cold column, where remote times are bandwidth physics no catalog changes — sync locally or add a persistent range cache (cache_httpfs) if that is your shape.

queryA per-modelB generic+viewsC normalizedD ducklake

The design answer: the generic-slot table with generated views is indistinguishable from per-model tables — within ~5% locally and ~1% over S3, on every query, cold and warm. Its 265-column NULL-padded schema costs nothing (Parquet encodes empty slots away; projection reads only aliased columns), and it adds new models with mapping rows instead of DDL. Pivot-at-query-time (C) is 20–35× off on risk-model workloads at this scale and survives only as the canonical source. FMP time-series access (16.7 s cold locally for one factor × 10 assets × 20 y) is the one dataset still wanting a dual materialization.

6The DRILL: "adding a model must never become a project"

The review's operational criterion, executed literally: two genuinely new config entries — AX_CA4_MH (small regional) and BARRA_EUE4_L_SFM1 (a Europe variant, ~10× the data) — added to the fleet and taken from config to queryable under each strategy, timed. Shared data-arrival (generating both models' 20-year normalized data): 12.5 min. Parity verified on all surfaces afterward.

strategyCA4 (small)EU variant (10×) structural change
A per-model74 s574 s +2 tables per layout, every model, forever
B generic-slot69 s + 0.5 s697 s + ~0 s rows into the existing table; slot-map rows + views regenerated (~0.5 s); zero new tables
C normalized0 s0 s nothing — but pays 8–500× on every query, forever (§5)

Materialization dominates and is unavoidable for either wide strategy; B pays ~20% extra write time on wide models (the 265-column schema is free to store, not free to write) and gets back a fleet whose object count never grows. Neither strategy touches existing data. The honest conclusion: adding a model is a 1–12 minute compute job either way — it only becomes "a project" through object sprawl and hand-maintained DDL, which are exactly what B's mapping table eliminates. Under DuckLake, the same operation becomes a transactional commit.

7The engine tier: query compute next to the data

The last mile of the remote story: client-observed cold can only match warm if the data stops crossing the ocean — so ship the SQL east and the Arrow result west. The tier is deliberately minimal and entirely open source: an ~80-line Python service (DuckDB + DuckLake + FastAPI, results as zstd-compressed Arrow IPC) on plain EC2, built from scratch by a boot script in ~3 minutes; a ~70-line client balancer (round-robin with a pre-query health ping; on any failure it re-reads a text file of engine addresses, pings all, and proceeds on the live set). No managed services, no proprietary engine, no load-balancer appliance — every component is Apache- or MIT-licensed and replaceable in an afternoon.

Two topologies were measured from New York against the Dublin data, same 16-vCPU budget: five small engines (r7g.large, ~0.75 Gbps NIC and a 10 GB cache each, $0.60/h fleet) versus one large engine (r7g.4xlarge, ~15 Gbps, one 90 GB shared cache, concurrent cursors, $0.97/h).

† the small-fleet anomaly: round-robin dispatched repeats onto engines that had never cached that query — five sharded 10 GB caches mean the fleet is only as warm as its coldest member, and each engine's 0.75 Gbps NIC makes its first GB-scale fetch cost ~15 s.

Findings. The engine tier achieves its purpose: client cold ≈ client warm on every query (CS1 1.70 vs 1.77 s; TS5 0.84 vs 0.50 s) because engine-cold is a boot event amortized across all clients, not a per-session tax — and on the big box even that boot event nearly vanishes (in-region fetch at 15 Gbps; DuckLake supplies the plan with zero discovery). Queries returning small results run interactively (0.3–0.8 s, floor = one transatlantic round trip + the health ping); queries returning 100 MB+ results are bounded by result transfer itself — the honest floor of any remote architecture. Between topologies the verdict is one-sided: one larger instance beats N smaller ones — 2× lower latency, 2.5× the throughput, no cache sharding, no fleet-warmth choreography, no balancer — at 1.6× the cost. Separate instances remain justified only for fault tolerance (two large boxes behind the same trivial balancer) or once a single NIC saturates (then r7gn, at ~4× bandwidth per size, before sharding). The trade-off the small fleet exposed is general: load balancing and cache locality pull in opposite directions — pooled caches resolve it; sharded ones require prewarming or affinity routing.

8Conclusion — the configuration, and why

Recommended configuration: one generic-slot wide table per layout (strategy B), dual-materialized date-major and asset-major, managed by a DuckLake catalog, over the normalized store as canonical source, with point-in-time served by version overlay. Every clause of that sentence is there because a measurement put it there:

clauseevidence
wide, dual-materialized (date-major + asset-major) wide beats pivot-at-query on all 12 queries, 2×–500× (§5); asset-sliced queries are structurally unprunable in any date-organized layout — locality is the only fix (§5 note)
generic-slot table, not per-model tables B ≈ A on every query, both locations, cold and warm (§5); adding a model = 0.5 s of metadata + zero new tables vs A's two tables per model per layout, forever (§6)
DuckLake catalog on top wins every warm remote cell — 76–125 ms from an ocean away, equal to local disk — by eliminating per-query S3 LIST/footer work (§5); turns add-a-model into a transactional commit (§6)
normalized store kept as canonical source zero-DDL landing zone for vendor data; feeds all materializations; carries the version dimension (§2, §6)
optional engine tier for remote-interactive use (one large box, or two for HA) client cold collapses to result-transfer time — 0.8–1.8 s for anything pruned, at any distance; one shared-cache box beats a sharded fleet 2× on latency, 2.5× on throughput (§7)
PIT via version overlay, not materialized versions as-of queries cost ~0.1–0.2 s over their non-PIT twins on wide layouts vs 26 s pivoted with arg_max (§5, CS5/TS6) — restatements are ~1% of dates, so materializing v2 everywhere would double storage for a corner case

Configurations discounted, and why

configurationwhy not
normalized-only (query the long store) 8–500× slower on risk-model workloads at global scale; 26 s PIT cross-sections; 42 s warm tall-thin scans; asset histories untenable (§5). Fine only for narrow date-sliced descriptor pulls.
per-model wide tables (strategy A) as the fleet surface identical query performance to B — so its only differentiator is cost: two new physical tables per model per layout, compounding across 10+ vendor models plus variants; schema churn on every model revision. The drill showed no offsetting speed advantage (§6).
one unified wide table with named factor columns rejected at design time (v1 plan): union of all models' factor names ⇒ NULL-riddled schema that churns on every revision, and cross-model column collisions (MOMENTUM ≠ MT_MOMENTUM) invite silent wrong-model reads — the failure mode B's generated views make loud.
single compromise layout (only date-major, or only asset-major) the wrong-layout penalty is ~12× locally and structural over S3 (asset-sliced queries against date-organized files read the whole store, §5 note); the second materialization costs ~19 s–2 h per model once, and repays it on every query.
materializing restated versions into the wide tables doubles wide storage for ~1% of dates; the overlay join achieves as-of correctness at ~0.2 s (§5). Revisit only if restatement rates are far higher than assumed (spec §6 question).
raw Parquet without a catalog (no DuckLake) workable locally with disciplined partition predicates, but remotely pays per-query LIST/footer round trips (~0.5–1.4 s warm floor, 3× cold penalty before the fix) and offers no transactions, snapshots, or atomic model adds (§5, §6).
typed generic pools (Cn_Int / Cn_VarChar / Cn_Float) the generalized form of B — unneeded here because factor payloads are uniformly DOUBLE; the float-only slot pool achieves the same add-a-model property with a simpler schema (§2). Becomes relevant only if heterogeneous attributes join the table.

Open items that could adjust this: practitioner validation of the proxy dimensions (spec §6), the FMP store's own dual materialization (§5), and a shared Postgres catalog for multi-writer DuckLake.

CREATE OR REPLACE VIEW v_cs_ax_ww4_mh AS
  SELECT year_month, cob_date, asset_id,
         F001 AS "MARKET_SENSITIVITY", F002 AS "MT_MOMENTUM", …,

Both wide strategies materialize version 1 only; point-in-time queries overlay normalized version-2 rows (measured at ~0.2 s over the non-PIT twin). Factor covariance is never re-materialized — all strategies share the normalized date-sorted table.
