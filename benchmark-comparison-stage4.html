Equity factor data API · Stage 4 · 2026-07-08

Same benchmark, four places: dev NVMe, EC2 NVMe, EC2 → S3, dev → S3

One grid (model BARRA_USE4_L, six queries, seven storage arms), four environments: this dev box on its own NVMe; r7gd.2xlarge (8 Graviton3 cores) on instance-store NVMe; the same instance reading Parquet from S3 in-region; and the dev box reading the same S3 data over the public internet — the remote-research pattern. Data regenerated on the instance from seed — byte-identical by construction.

Moving to EC2 costs a flat 1.70× warm (median across all cells) — core count, nothing structural: every layout conclusion from the local benchmark holds unchanged.

Moving the data from instance NVMe to in-region S3 costs +9% warm at the median. DuckLake is flat — ≈1.0× — because its catalog already knows every file and byte range; plain Parquet reading pays up to 4.5× on small pruned queries, which are round-trip-bound.

Cold is where S3 bites: 0.3–0.5 s of connection and footer discovery before the first byte, and unpruned scans stream the whole store at ~400 MB/s — the worst cold cell from inside AWS is 5.6 s.

From the dev box over the internet, DuckLake warm queries run at 14–27 ms — indistinguishable from local NVMe. Plain Parquet reading pays ~30× for the same queries (0.3–0.8 s): S3 listing and footer round trips at internet latency, exactly the degradation the catalog exists to remove. Cold full-store scans cost ~50 s at this downlink — the case for syncing, or for never querying the wrong layout remotely.
