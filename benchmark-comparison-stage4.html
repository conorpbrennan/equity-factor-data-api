Same benchmark, four places — ending at a New York laptop querying AWS Ireland

One grid (model BARRA_USE4_L, six queries, seven storage arms), four environments: the New York dev box on its own NVMe; r7gd.2xlarge (8 Graviton3 cores) on instance-store NVMe in eu-west-1 (Dublin); the same instance reading Parquet from S3 in-region; and the New York box reading the same Dublin bucket across the Atlantic — ~75 ms of ocean per round trip, the remote-research pattern at its least forgiving. Data regenerated on the instance from seed — byte-identical by construction.

Moving to EC2 costs a flat 1.70× warm (median across all cells) — core count, nothing structural: every layout conclusion from the local benchmark holds unchanged.

Moving the data from instance NVMe to in-region S3 costs +9% warm at the median. DuckLake is flat — ≈1.0× — because its catalog already knows every file and byte range; plain Parquet reading pays up to 4.5× on small pruned queries, which are round-trip-bound.

Cold is where S3 bites: 0.3–0.5 s of connection and footer discovery before the first byte, and unpruned scans stream the whole store at ~400 MB/s — the worst cold cell from inside AWS is 5.6 s.

From New York to Dublin — an ocean away — DuckLake warm queries run at 14–27 ms: indistinguishable from the laptop's own NVMe. Plain Parquet reading pays ~30× for the same queries (0.3–0.8 s), because S3 listing and per-file footer reads each cost a transatlantic round trip — exactly the degradation the catalog exists to remove. Cold full-store scans cost ~50 s at this downlink — the case for syncing, or for never querying the wrong layout remotely.

The verdict. Best performer overall: DuckLake over the dual wide transforms on S3. It is the only arm that is fast in every environment — within ~1.3× of the unbeatable-but-local native tables at home, and flat all the way out to a laptop an ocean from the data. Direction: keep both wide layouts in DuckLake on S3, route each query to its native layout, distribute the 4 MiB catalog to clients (or move it to a shared Postgres), and never brute-force a wrong-layout scan from outside AWS. And the cold/warm split below adds the one qualifier: if cold-start latency matters, sync the two wide tables to the client and query locally — warm is identical either way. Native DuckDB tables remain the right cache for a single hot workstation; the normalized store is an archive format, not a query surface.

Warm, location is irrelevant: the remote filled marks sit on top of the local ones (14–27 ms either way — DuckLake plans locally and DuckDB caches the ranges). Cold, location is everything: ~2.7–5.4 s remote against ~65–110 ms local — a ~40× gap that no catalog can remove, because it is TLS setup and transatlantic round trips. So the operating rule falls out directly: if cold-start latency matters — CI jobs, fresh kernels, the first query of the day — sync the two wide tables to the client (~2.7 GB compressed for this model, a one-off few minutes; the daily delta is one partition per layout) and query locally. Warm performance is identical either way, so syncing costs nothing except disk. The lazy alternative, now measured: the cache_httpfs extension persists fetched ranges to local disk, and a fresh process whose ranges any earlier session touched starts at 24–83 ms first-touch (sample client --cache; 376 MB on disk after the full two-model suite) — sync's steady state, warmed per-range on demand instead of 2.7 GB upfront.

Bytes-read accounting covers the NVMe grids only (httpfs traffic arrives over sockets, invisible to rchar); the Stage-3 pruning figures — megabytes for right-layout queries, the full ~1.4 GB store for wrong-layout ones — describe what S3 serves, since the plans are identical.

The warm cache, quantified. Warm figures are DuckDB's external file cache: fetched S3 byte ranges are kept in the buffer pool (enable_external_file_cache, on by default since 1.3) and capped by memory_limit — 80% of RAM unless set (49.9 GiB on the NY box), evicted LRU alongside query working memory, so a long session can hold the entire 2.7 GB hot dataset. Inspect it with SELECT * FROM duckdb_external_file_cache(); flush it with SET enable_external_file_cache = false, which drops every cached range immediately (verified: queries return to network cost; re-enabling repopulates). The cache is per-process and not persistent — a fresh session always starts cold, which is exactly what the sync option or the cache_httpfs extension addresses (the latter measured at 24–83 ms fresh-process first-touch; see the key view above).
