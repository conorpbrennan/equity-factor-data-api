Equity factor data API · Caching design · 2026-07-20

Caching: the working set we have, the ideas on the table, and where record & playback fits

This note does three things. It describes, in full, the caching that exists in the prototype today and the measurements that motivate it (§1–§2). It records Chris's design input from the 14 July review email and subsequent discussion (§3). And it lays out a proposal that has not been in the conversation yet: a record & playback layer at the data-sourcing boundary, where caching and regression testing turn out to be the same mechanism (§4). §5 shows how the three shapes compose rather than compete, and §6 closes with a recommendation on what to build next and what to defer.

1What the prototype does today — a working-set cache

The prototype's cache (python_src/modelfacade/cache.py) is a working-set cache, not query-result caching. The distinction drives everything: rather than memoizing individual queries, the user declares a predictable working set once — a date range and an asset list per dataset — and every subsequent request that falls inside that coverage is answered from memory. Anything outside falls through to the loader untouched.

1.1  Layering — where the cache sits

Caching lives strictly at the facade layer, so leniency never leaks into core. The stack, bottom-up:

  • DataSource (datasource.py) — a @runtime_checkable Protocol owned by the core layer. Requests are semantic and model-agnostic — which fact, which dates, which assets — never SQL, globs, or layout. Four methods: dim(), read_fact(), has_column(), date_bounds().
  • Store (store.py) — implements DataSource; routes s3:// roots to the in-region query engine, scans locally otherwise.
  • Model (core.py) — the strict core. Takes a DataSource; never loads directly, never imports the facade.
  • ModelFacade (facade.py) — composes a source, hands it to Model, and holds the cache. Every read method wraps the core call in a loader lambda: self.cache.get(dataset, lo, hi, ids, loader).

1.2  The three classes

classrolebehaviour
Coveragedeclaration start, end, frozenset of assets (None = all) per dataset
NoCachethe default get() always runs the loader; put() raises rather than silently no-op; stats all zero
UserCachethe opt-in coverage test, filtered serve on hit, fall-through on miss, hit/miss counters, clear(), parquet persistence

Caching is off by default (settled 2026-07-15). The facade constructor does self.cache = cache if cache is not None else NoCache() — the speed-versus-staleness trade-off is always the caller's, never a default. The coverage test is containment: the cached range must contain the requested range and the requested assets must be a subset of covered assets. A hit filters the warmed frame by COB_DATE.is_between(start, end) and asset membership; a miss runs the loader in full.

1.3  Warm, persist, cold-start — the daily lifecycle

cache = UserCache()
mf = ModelFacade.load("AX_WW4_MH", cache=cache)

mf.warm(assets)          # YTD loadings + specific risk for the book,
                         # all factor returns (official stream)
mf.save_cache()          # parquet-per-dataset + manifest.json under
                         # <base>/usercache/<as_of>/<model_id>/

# next morning, or offline on the train:
mf = ModelFacade.from_cache("AX_WW4_MH")   # never touches the store
mf.load_cache(max_age_days=1.0)            # 1-day TTL, opt out with None
cache.clear()                              # known restatement: drop the set, re-warm
  • warm(assets, as_of, sec_id_type) loads year-to-date loadings and specific risk for the given assets plus all factor returns; raises if the cache is a NoCache — a warm with nowhere to live is a bug, not a no-op.
  • save_cache() also persists the dimension tables (model_master, factor_master, asset_xref), so a later session can rebuild the Model with no store at all.
  • from_cache() starts a session entirely from the persisted set — offline cold start. 'latest' resolves to the set's as-of date.
  • The 1-day TTL (max_age_days=1.0) enforces the morning re-warm cadence; max_age_days=None is the explicit opt-out for knowingly using an older set.
  • clear() invalidates the whole working set but keeps the session hit/miss counters — the counters describe the session, not the set.
  • warm_cache.py is the cron-ready morning job: repeatable --model, a positions file, per-model isolation (one model failing doesn't stop the rest).
  • T0 estimates bypass the cache entirely (get_factor_returns(estimates=True)) — their whole value is freshness.
  • Cache base is $FACTOR_CACHE_DIR or the system temp dir — temp by default, so unconfigured working sets self-expire; set the env var to make them persistent.

1.4  What the TTL is — and is not

The 1-day TTL is a staleness guard, not an incremental update. When save_cache() writes a set, it stamps saved_at (UTC) in manifest.json. When a later session adopts the set via load_cache() or from_cache(), the facade compares now − saved_at against max_age_days and raises a ValueError if the set is older — it cannot silently hand you Tuesday's data on Thursday. max_age_days=None skips the check for when an older set is knowingly acceptable (offline for a few days with last Friday's snapshot).

The next-day load does not extend the previous set by one day — the set is rebuilt from scratch. The morning job calls warm() again, which re-pulls the full working set from the store (YTD loadings and specific risk for the assets, all factor returns), and save_cache() writes a fresh set under a new as-of directory. Yesterday's set is simply abandoned. "1-day TTL + morning re-warm" therefore means: every day gets a complete fresh snapshot, and the TTL only stops you accidentally consuming a stale one. The cost is acceptable because the warm is a bulk sequential pull — the cheap kind of query.

The incremental version — load yesterday's set, pull only the new COB date, extend coverage by one day — is exactly Chris's fetch-missing-days-and-merge idea (§3.2), and it is not built. It would make the morning job much cheaper, but note the correctness trap it must clear first: an append-only warm never picks up restatements to earlier dates, whereas the full daily rebuild gets them for free. A delta-warm therefore needs the restatement story to be solid before it ships — as_of/status in the cache key, and clear() on known restatements as the escape hatch.

One known gap. The persisted-set path is keyed by (as-of date, model_id), but the in-memory key is dataset name + coverage only — nothing in it records which as-of view a frame was loaded under, or whether it is official data versus a T0 estimate. That is safe today only because a cache instance ever holds one view: warm() loads latest-official, and estimates bypass the cache entirely.

Worked example of the failure. Loadings for March 15 are restated by the vendor on April 2. You warm on April 10, so the cached frame holds the restated March 15 values — that is what "latest" means on April 10. Someone then asks the point-in-time question: March 15 loadings as known on March 20 (as_of="2026-03-20"). The cache checks its key: dataset "factor_loading"? Yes. March 15 inside the covered range? Yes. Hit — and it serves the restated values, when the correct PIT answer is the pre-restatement ones. Silently wrong, dressed as a fast success: the coverage test cannot catch it, because the requested date genuinely is covered — what differs is the as-of view, and that is not part of the key.

The fix: make the view part of the identity. The cache key must grow as_of/status the moment restatement queries are exposed — then the PIT query above is a clean miss that falls through to the loader, which is the correct behaviour. This is a firm requirement whichever cache design wins, because every shape has the same hole if identity is incomplete: the working set's coverage key, the decorator's __cache_identity__ (Chris's proposal lists model, variant, as_of, status explicitly), and the SHA-256 request hash in §4 — a request is not fully specified without as_of, so it is not replayable without it.

2The numbers — why a cache, and why a working set

The 20-year, twelve-model store is 680 GB; an interactive working set is small. That asymmetry is the whole premise. The three-way storage benchmark (AX_WW4_MH: 248 factors, 58,000 coverage assets, 302M wide rows) shows warmed local answers are already sub-second — and the local-vs-AWS comparison shows what happens without a cache when the data is remote.

Reading of the numbers. Cold trans-Atlantic scans are 145–159× local NVMe. The working-set cache converts that class of query — the repeated, predictable, book-shaped pulls that dominate an interactive session — into local sub-second answers, and the persisted set makes the first session of the day warm too. The engine handles the long tail the set doesn't cover.

3Chris's input

3.1  Invalidation policy (email, 14 July)

Pre-warmed caches with a TTL of 1 day — refresh every morning — plus a function to invalidate the entire cache (cache.clear()). Restatements are infrequent, so a same-day cache is fine in most cases; a use case that needs fresher data bypasses the cache; a known restatement clears and re-warms. Status: all implemented — the TTL, the clear, the bypass (estimates), and the morning warm job.

3.2  Cache semantics

  • Opt-in, caller-controlled — off by default; the caller owns the speed-versus-freshness trade-off. Implemented.
  • Key includes model, variant, as_of, status — a point-in-time query never gets a later restatement; T0 estimates are never served as official data. Partly implemented: estimates are excluded by bypass, and the persisted path carries (as_of, model); as_of/status in the in-memory key is the open item from §1.
  • Date-range requests fetch only the missing days and merge them in. Not implemented: today's coverage test is all-or-nothing — a request partly outside coverage falls through in full. Fetch-and-merge is a natural extension of the coverage model.

3.3  Reusable design

Chris's preferred shape is a reusable library: any data-access class becomes cache-aware by accepting a cache= argument, defining __cache_identity__(), and decorating methods with @cached_dataframe(date_range=True). The stated alternative — build the cache directly into ModelFacade — is a smaller lift but not reusable. The open question he posed: is there value in having it generically available?

One distinction worth keeping in view: the decorator and the working set cache different things. The decorator memoizes query results; the working set declares coverage — which is what makes the persisted morning warm and the offline cold start possible. Current lean: make the opt-in cache= convention firm-wide now, build the generic layer when a second data class needs it (a mechanical refactor).

Two practical considerations are worth weighing when the time comes. Debuggability: a per-query memo is only as correct as its key, and every decorated method is a fresh opportunity for an identity bug — one argument missing from __cache_identity__ and the memo returns silently wrong data with nothing visible at the call site. The working set fails safer: coverage is declarative and inspectable (stats, the coverage dict, manifest.json on disk), and a miss just runs the loader — the failure mode is slowness, not wrongness. Lifetime: an in-memory memo dies with the process, and making it persistent means rebuilding exactly the persistence machinery the working set already has. Residual value: with a warm working set, covered queries are already served in-memory sub-second — memoization's marginal win is repeated identical uncovered queries and metadata pulls, and record & playback (§4) serves that case at the boundary anyway: on disk, across instances, auditable.

4Proposal — record & playback: caching and regression testing are one mechanism

There is an intersection between caching and regression testing that neither design above exploits. The pattern, used successfully in past systems: enable a record & playback capability at the data-sourcing boundary. The record is a cache — but it is a disk cache, in parquet, keyed by the request itself, and its second life is as a regression fixture.

4.1  Mechanics

  • Every data-sourcing request is expressible as a canonical JSON document — which implies (and enforces) a JSON schema for requests. This is a feature, not overhead: it makes the request surface explicit and versionable.
  • The canonical JSON string is hashed with SHA-256; the digest is the key.
  • Two files per request: <key>.request (the JSON, kept human-readable for audit and debugging) and <key>.response (the result, parquet).
  • Three modes: record (create the cache on first pull), force re-record (recreate when the upstream data or the schema changes), and playback (subsequent reruns served entirely from disk).
  • The store is persistent across client instances when configured as such — the same property the persisted working set already has, generalized to arbitrary requests.
# request → canonical JSON → SHA-256 → paired files
{"fact": "factor_loading", "model_id": "AX_WW4_MH",
 "start": "2026-01-01", "end": "2026-06-30",
 "assets": [1017, 2044, ...], "version": 1, "as_of": "2026-06-30"}

  ⇒ 9f2a…c41e.request   (the JSON above)
  ⇒ 9f2a…c41e.response  (parquet)

4.2  The regression payoff

Once a session's pulls are recorded, the pairs are captured inputs for regression tests. A model-based regression run replays from the recorded set and is therefore dependent on no data infrastructure at all — no Snowflake, no S3, no query engine, no network. That gives:

  • Deterministic, repeatable inputs — a regression failure is a code change, never a data drift ambiguity.
  • CI that runs anywhere the parquet fixtures are, at local-disk speed (§2: sub-second vs up to 159× slower remote).
  • Force re-record as the deliberate, auditable act of refreshing the baseline — the diff of the .request files documents exactly what changed.

4.3  DataInterface — one choke point for all sourcing

To get this in one place across all data-sourcing calls, the record/playback layer wraps a single DataInterface at the sourcing boundary. The prototype is already shaped for this: the core-owned DataSource protocol (§1.1) is exactly that choke point — semantic requests, no SQL, no layout. A recording source is just a DataSource that wraps another DataSource: 

Model ── DataSource protocol ── RecordingSource(mode=record|playback|force)
                                      └── Store (local | s3 → query engine)

Because requests are already serialized as canonical JSON, the same interface is wire-ready: pointing sourcing at a server-side process later is a transport change, not a redesign. The JSON request schema written for hashing is the API contract the server would accept; the parquet response is the payload it would return. Record & playback today, client–server tomorrow, same seam.

4.4  Fixture housing — a separate repo, fixed dates only

Recorded fixtures live in their own git repository, holding only fixed-date request/response pairs for regression testing. This carries one canonicalization rule with it: recorded requests contain absolute dates only — 'latest' or any relative as-of resolves to concrete dates before hashing, otherwise the same logical request hashes differently every day and replay is not deterministic. The consequences fall out cleanly:

  • The main repo stays lean; fixtures change only on a deliberate re-record, a different cadence from code.
  • A baseline refresh (force re-record) is a reviewable PR in the fixture repo — the diff of the .request files documents exactly what changed and why.
  • CI pins the fixture repo by commit, so a regression run is fully specified by (code SHA, fixture SHA).
  • Fixed-date slices are small — a regression suite's pulls, not the 680 GB store — so plain git suffices; no LFS needed initially.
  • Data only. The fixture repo holds request/response pairs and nothing else; the replay harness (the RecordingSource and the test wiring that maps fixtures into playback mode) lives in the main repo beside datasource.py. Fixtures stay a pure, tool-agnostic dataset; harness changes version with the code they test.

5Synthesis — three shapes, one seam, shared requirements

Working setdeclared coverage per dataset facademorning warm, offline cold start, covered-subset serving built

These are complementary, not competing. The working set serves the interactive session; record & playback serves determinism and testing at the boundary below it; the decorator is the generalization to adopt when a second data class asks for caching. Notably, record & playback is request-keyed memoization — Chris's decorator semantics — placed at the sourcing seam, where it additionally buys the regression capability and the future server path.

Shared requirements, whichever shapes ship.

· Opt-in, off by default — settled, keep it.

· A canonical, versioned JSON request schema — required for the SHA-256 key, and the future service contract for free.

· as_of/status in the cache identity everywhere — the working-set key gap (§1) and the request hash both need it; PIT answers must never see a later republication, and T0 estimates must never replay as official.

· Explicit invalidation as a deliberate act — clear() + re-warm for the working set, force re-record for the fixture store; both auditable.

6Recommendation

Record & playback is the strongest addition on the table — but its value is as a determinism and testing mechanism, not as a rival cache. Exact-match request hashing is the wrong shape for an interactive cache (reorder the asset list or extend the range by a day and you miss — precisely the case the working set's coverage model handles), and exactly the right shape for a regression contract, where byte-identity is the feature. Keep both layers, each doing the one thing it is good at; resist unifying them.

6.1  Build next

  • RecordingSource wrapping any DataSource, with the three modes — and strict playback: in playback mode a miss raises, it never falls through to the store. A silent fall-through quietly reintroduces the infrastructure dependency the mode exists to remove; a hard failure says honestly that the code's data demands changed and the baseline needs a deliberate re-record.
  • One model regression wired to fixtures end-to-end as the proof — record a session, replay it in CI with no store, no engine, no network.
  • as_of/status into both identities while there. A request is not replayable unless its identity is complete, so record & playback is a natural forcing function for the PIT-correctness requirement in §1 — close the working-set key gap in the same change.

6.2  Where the real work is: canonicalization

The hash is trivial; the canonical form is not. Sorted keys, sorted asset lists, one date format, and — the subtle one — defaults materialized rather than omitted, or {"version": 1} and {} become two different fixtures for the same pull. Absolute dates only in recorded requests — 'latest' resolves before hashing (§4.4). Define the request as a typed model that normalizes before serializing, and put a schema version inside the hashed payload so a schema change deliberately invalidates the fixture set instead of silently colliding with it.

6.3  Cautions

  • Fixture sprawl — addressed by §4.4. The separate fixed-date fixture repo bounds growth: fixtures are a regression suite's slices, not accumulated session traffic, and grow only on deliberate re-record. Against real vendor data at SFM, recorded fixtures still carry licensing weight — worth a sentence in any proposal.
  • Don't headline the server path. The JSON request schema being wire-ready is real, but an actual service brings auth and versioning decisions not worth defending yet. Keep it as the footnote it is in §4.3.

6.4  Defer

The missing-days fetch-and-merge (§3.2) and the generic @cached_dataframe layer (§3.3) both wait until something concrete demands them — the merge until partial-coverage misses show up in practice (and not before the restatement story is solid — see the append-only trap in §1.4), the decorator until a second data class asks for caching. Both are mechanical extensions of seams that already exist; neither blocks anything above.

Positioning note. Record & playback sidesteps the decorator-versus-facade ownership debate entirely — it is neither, it sits at the core-owned boundary — and it gives QDS a deterministic test harness for FMA's own regression, which strengthens the proposal whoever ends up owning ModelFacade.
