Equity factor data API · package documentation · v2 report

conventions & modelfacade — the user-layer packages

Two packages, one dependency arrow. conventions makes the firm-wide data conventions importable — shared column constants, executable unit conversions, typed identifier schemes, canonical argument names — so a convention is something code references rather than a document code drifts from. modelfacade is its first consumer: a strict core Model and a lenient user-facing ModelFacade over the factor store, importing every literal, scale, and scheme from conventions. Together they scaffold the first two roadmap projects and demonstrate the intended pattern for everything after.

Part I · conventions

1Why a library, not a document

Inconsistent conventions create friction at every seam: time lost renaming columns, converting units, and remembering which spelling of as_of this particular library wanted — and each inconsistency is a place for subtle bugs to hide. Documents that state the conventions exist and disagree with each other. The fix is a library that is the conventions: new code imports them, and legacy data is sanitized by the adapter toolkit at the point where it enters the system. Target-plus-adapter, not forced migration.

moduledecisionstraw man
columnscolumn-name case + shared constants snake_case; one string constant per column name, imported everywhere, never re-typed. Toolkit: snake_case(), rename_snake().
identifierssecurity identifier schema sec_id + sec_id_type for single-scheme frames; explicit sec_id_<scheme> columns (via sec_id_col()) for multi-scheme; a closed enum for scheme values.
unitsscale and frequency per quantity returns daily decimal; volatility annualized decimal; covariance annualized decimal²; money in millions USD. All conversions executable via scale_to_canonical().
signaturesfunction argument naming as_of / start / end / assets / factors / model; plus a DISCOURAGED map of spellings seen in the wild.

The consistency has more value than any particular choice: every decision here is a straw man, and swapping one is a one-line change every importer picks up.

2Usage

Column constants — reference, never re-type

from conventions import COB_DATE, ASSET_ID, FACTOR_ID, VALUE

df.filter(pl.col(COB_DATE) == as_of)        # not pl.col("cob_date")

Adapter toolkit — sanitize legacy data at the boundary

from conventions import snake_case, rename_snake

snake_case("CobDate")      # 'cob_date'   (PascalCase)
snake_case("COB-DATE")     # 'cob_date'   (upper + hyphen)
snake_case("ASOFDATE")     # 'asofdate'   (all-caps run stays one word)

df = rename_snake(df)      # every column -> snake_case (polars or pandas)

Units — conversions are multipliers, applied once at the boundary

from conventions import scale_to_canonical

scale_to_canonical("specific_risk", "ann_vol_pct")   # 0.01
scale_to_canonical("covariance", "daily_var")        # 252.0
scale_to_canonical("return", "weekly_bps")           # ValueError, loudly

df.with_columns(pl.col(VALUE) * scale_to_canonical(kind, vendor_convention))

Identifiers — typed schemes, no string drift

from conventions import SecurityIDType, sec_id_col

SecurityIDType.AXIOMA          # 'AXIOMA' — matches asset_xref.vendor
sec_id_col(SecurityIDType.BARRA)   # 'sec_id_barra' (multi-scheme frames)

Identifier mapping rules (stated once, enforced in review)

Vendor↔vendor mapping is many-to-many — one Barra id can map to several Axioma ids and vice versa. Mapping tables must be dated. Vendor→vendor is never chained silently: always via the internal asset_id.

Part II · modelfacade

4Strict core, lenient user layer

Core model libraries are (rightly) strict: exact date types, native identifiers, validated inputs, always-right-or-fails. End users need the opposite: string dates, whatever identifier scheme their positions carry, one-liners, discoverability, and caching they control. Serving both from one class satisfies neither. The resolution is two layers with two contracts and explicit conversion between them — and because they are separate objects, user-cache leniency can never leak into a core computation.

Model (core.py) — strict

from datetime import date
from modelfacade import Model, Store

m = Model(Store.open(root), "AX_WW4_MH")
m.factor_loadings(date(2025, 6, 30),
                  assets=[101, 102])
m.factor_loadings("2025-06-30")
# TypeError: core does not coerce

datetime.date only (datetimes rejected too — a COB has no time) · internal int asset ids only · factor ids validated against factor_master · values raw, exactly as the vendor published; conventions exposed as metadata via m.conventions.

ModelFacade (facade.py) — lenient

from modelfacade import ModelFacade

fac = ModelFacade.load("AX_WW4_MH")
fac.get_factor_loadings("latest")
fac.get_specific_risk("2025-06-30",
    assets=["AX01234567"])   # vendor ids ok
fac.core                     # strict layer back

String dates + 'latest' · vendor ids resolved via asset_xref · canonical units on the way out · wide one-liners · pre-warmable user cache · output="pandas" if that's what your notebook speaks.

Store root comes from $FACTOR_STORE_ROOT or load(..., root=...); local paths and s3:// both work.

Publication streams, side by side

Factor returns carry two publication streams in the same table — the vendor official and the same-day T0 estimate — as ordinary rows discriminated by the type column. type answers "which stream?"; version_id answers "as known when?" (restatements). Keeping them orthogonal means an official row can itself be restated (type=OFFICIAL, version_id=2), and the stream toggle is an equality filter — never a join: 

fac.get_factor_returns()                  # official stream (the default)
# cob_date     factor_id   value   type         version_id
# 2025-01-15   IND01       0.005   OFFICIAL     1

fac.get_factor_returns(estimates=True)    # same one-liner, estimate stream
# 2025-01-15   IND01       0.006   T0_ESTIMATE  1
#   — bypasses the cache: an estimate's whole value is freshness

Existing stores predate the column; nothing about them breaks and no backfill is required before adopting this:

requeston a store without the type column
official (the default) Works unchanged — a legacy store's rows are the official stream, so serving them unfiltered is correct, not degraded.
estimates=True Refuses loudly ("factor_return has no type column — this store carries no estimate stream") — never silently returns official numbers to a caller who asked for estimates.

The selftest's micro store deliberately contains one of each: a model whose factor_return carries both streams, and a legacy one without the column.

6The pre-warm cache

Not query-result caching: the premise is that most of a user's questions hit a predictable working set — year-to-date data for the assets they hold, in the model they care about. warm() loads that set once; any request covered by it (subset dates, subset assets) is served from memory, and anything outside falls through to the store and still returns correct data. 

fac = ModelFacade.load("AX_WW4_MH")
fac.warm(position_asset_ids)                    # YTD loadings + specific risk
                                                # for holdings, all factor returns
fac.get_factor_loadings("2025-06-15", assets=held)   # served from cache
fac.get_specific_risk("latest", assets=held)         # served from cache
fac.cache.stats                                 # {'hits': …, 'misses': …, 'rows': …}

fac.save_cache()                     # persist: parquet + manifest, keyed by
                                     # (as-of date, model) under
                                     # <temp>/usercache/<as_of>/<model_id>/
# --- a later session, same machine ---
fac = ModelFacade.load("AX_WW4_MH")
fac.load_cache()                     # newest saved set for this model;
                                     # as_of= pins a date; refuses another
                                     # model's set

Design intent

Pre-warm the expected set in the morning, before anyone asks — most of a risk manager's day then runs from memory, and the trade-off between freshness and speed stays in the user's hands rather than in a TTL nobody chose. Persistence extends the same idea across sessions: one scheduled job warms and saves, every later session loads and starts hot. The saved set is frozen as of the save — re-warming is the caller's decision, not a TTL's. Once the backing store is fast enough the cache becomes unnecessary without any API change. The scheduled job exists: warm_cache.py reads a positions file, warms any number of models (one failing doesn't stop the rest), persists each set, and exits nonzero on any failure — cron-ready; try it with python warm_cache.py --demo.

Part III · verification & open questions

8Verification

python -m modelfacade selftest fabricates a complete micro store in a temp dir (genv2 layout, deterministic values) and asserts every contract on this page — 12 checks, one PASS line each, covering both packages: the conventions toolkit and unit multipliers, core strictness, sparse reads, date and identifier leniency, exact unit conversions (29.0 ann_vol_pct → 0.29), the T0 estimate stream (both publication streams, cache bypass, legacy-store fallback), wide-pivot fill, cache hit/miss accounting, cache persistence across sessions, pandas at the return boundary, and the wrap/unwrap round trip. python -m modelfacade demo --root DIR runs the one-liner story against a real v2 store.

9Open questions

conventions

  • Who ratifies? These need a single owner with the standing to make them stick; a library can enforce spellings, not adoption.
  • Closed enum or free string for sec_id_type? Closed here (drift-proof), but that makes adding a scheme a code change.

modelfacade

  • Which dataframe library does the user layer speak? The installed base of risk notebooks is pandas-native, which argues for output="pandas" as the deployed default; this repo defaults to polars to match its own stack. The mechanism makes the default the only real decision.
  • Ownership. Facade and core must stay in sync, which argues for the core library's owners; the alternative is the team owning application-to-portfolio use. Code location next to the core either way.
  • Units at which layer. Core-raw / facade-canonical (as built) means two callers can hold different numbers for the same quantity; core-canonical would push vendor conventions down into ingest.
  • Cache staleness under restatement. Sharper now that working sets persist across sessions (a loaded set may be days old). Pre-warming sidesteps TTLs, but a warmed frame can go stale within a day when a restatement publishes — is re-warm-on-publication enough?
  • Point-in-time. version is plumbed through the core; the facade doesn't yet expose as_known_on= overlay queries (the store supports it — see the v2 report, PIT section).
  • Fast path. The facade currently reads the normalized store and pivots — correct anywhere, slow at global scale; wiring the generic-slot tables (Store.generic_cs_glob) into get_factor_loadings is the natural next increment.
